# RT8 — Vacuous denial-path test (status asserted, mutation not)

This file contains the full normative procedure for `RT8`. The compact row in `../common-rules.md` is only its routing summary.

## Full rule

| ID | Pattern | Procedure | Severity if missed |
|---|---|---|---|
| RT8 | Vacuous denial-path test (status asserted, mutation not) | A test that exercises a gate's *denial* path — authorization/step-up reject (403), rate-limit reject (429), fail-closed (503), permission-denied — MUST assert BOTH the denial status AND that the guarded operation did **not** execute. A test asserting only the status is vacuously green: delete the gate and let the mutation proceed, and the test still passes because nothing checks the side effect. This is a false-green on exactly the security control the test advertises. **Required pattern** (illustrative — adapt to framework): `expect(res.status).toBe(403)` (existing) PLUS `expect(mutationSpy).not.toHaveBeenCalled()` (the guard) — or `toHaveBeenCalledTimes(0)` on the write/create/delete/revoke spy. The negative assertion names the precondition the status assertion silently depends on; without it, a green test cannot distinguish "gate blocked the mutation" from "mutation never ran for an unrelated reason (wrong fixture, missing setup, mocked-away handler)". Apply to: authz/authn guards, step-up re-auth, rate limiters, fail-closed degradation, idempotency rejects, any test whose name implies "denied request does not mutate". **Distinct from RT4** (race cardinality) and **RT7** (gate must be red-capable): RT8 is the assertion-completeness of a single non-race denial test — RT4 needs both race branches to occur, RT7 proves the gate *can* fail, RT8 ensures the test *observes the side-effect's absence*. **Mechanical detection**: `bash ~/.claude/hooks/check-vacuous-denial.sh [base-ref]` walks Jest / Vitest `it(...)` / `test(...)` blocks per changed test file (paren-balance tracking), flags when a block asserts a denial status (`.toBe(403/429/503)` on any receiver) AND the file declares a mutation-verb spy (`vi.fn()`/`jest.fn()` named create/update/delete/deleteMany/upsert/insert/save/revoke/...) AND has NO negative assertion (`.not.toHaveBeenCalled(` / `.toHaveBeenCalledTimes(0)`). The denial-status line must be in the diff `+` set to flag. v1 covers `it`/`test` blocks in TS / JS test files; pytest / Go / RSpec out of scope. The hook can't classify whether the gate is a security control — reviewers escalate to Critical manually based on the gate's role. Project-specific mutation verbs / status codes extend via `EXTRA_MUTATION_VERB_RE` / `EXTRA_DENIAL_STATUS_RE`. | Major (Critical when the gate is a security control — authz/authn/rate-limit/step-up/fail-closed) |
