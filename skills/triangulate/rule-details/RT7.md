# RT7 — New guard / test / gate must be proven able to fail

This file contains the full normative procedure for `RT7`. The compact row in `../common-rules.md` is only its routing summary.

## Full rule

| ID | Pattern | Procedure | Severity if missed |
|---|---|---|---|
| RT7 | New guard / test / gate must be proven able to fail | Any newly added regression test, security gate, or invariant check MUST be demonstrated capable of going *red* for the case it claims to cover — by injecting the violation (mutation) or reverting the fix and confirming the test/gate fails. A green check that *cannot* fail gives false assurance on exactly the invariant it advertises. Three recurring blind shapes: **(a) restriction / narrowing tests** — when a change REMOVES a value from an allowlist or TIGHTENS an acceptance predicate (drops a weak algorithm, a legacy label, a permissive branch), additive coverage stays green on revert because nothing exercises the now-rejected input. Ship a negative test that feeds the now-rejected input and *fails if the narrowing is reverted*. **(b) authored-but-ungated detector** — a check script / lint / policy grep exists but has zero callers in the gate the project treats as authoritative (CI / pre-push), or runs in only one of local-vs-CI surfaces; it reports PASS by never executing. Verify every authored check is actually invoked by the authoritative gate, and that local and CI run the same set. **"Same set" means same command AND same environment**: a check invoked in both local and CI can still diverge if the two environments differ — the local run carries generated artifacts (ORM/RPC codegen, build output) a "static"/"no-generate" CI job omits, so an identical command passes locally and fails (or vacuously passes) in CI. Confirming the caller exists is necessary but not sufficient; for a check added to a generate-skipping job, also apply the generated-artifact-independence sub-check (displace the artifact, re-run — see triangulate phase-2 Step 2-1 item 7). **(c) structurally-blind gate** — a wired gate cannot fail because its fixture / precondition is incomplete: golden vectors omit a legal variant of the thing forbidden; a diff-based guard whose base ref errors exits 0 (fail-open) instead of failing; a presence assertion (`expect(el).toBeVisible()`) that stays green on an ordering / layout / a11y-attribute regression — assert *order / identity / the literal attribute*, not mere presence. Three further blind shapes (mined from multi-round gate-hardening reviews): **(d) red-proof rationale rewording** — editing or re-attributing the *stated reason* an existing test can go red is itself a red-proof claim: re-execute the specific named mutant (on a scratch copy) and observe the failure BEFORE the reworded sentence lands. This class recurred three times in a single review thread precisely because RT7 compliance was checked only for new rows, not for prose edits about existing ones. **(e) gate matcher availability** — regex/pattern-matching logic inside a gate is production attack surface: a widened detection regex can introduce catastrophic backtracking (ReDoS) so the guard *hangs* rather than misclassifies. Every regex-based gate ships a pathological-input timing self-test (long repeated-pattern string completes fast), in addition to positive/negative classification fixtures. **(f) AST-classifier authoring checklist** — a gate that classifies CODE (is this a real test/guard/caller?) must climb the full binding ladder up front, not through adversarial rounds: text match < AST node existence < import-symbol binding < execution binding < framework binding. Concretely: (i) bind every matched identifier to its import-origin symbol (`(aliasNode ?? nameNode).getSymbol()` vs callee symbol) — never bare name text, which shadowing locals and aliases both defeat; (ii) verify the matched node EXECUTES — nearest enclosing function is a callback of a non-skipped, framework-symbol-bound registration — not merely exists (calls parked in uninvoked functions, top level, or skipped suites never run in CI); (iii) accepted modifier/variant forms are an ALLOWLIST (unknown future APIs default to not-counted, fail-closed) — never a denylist of known-bad forms; (iv) derive case coverage from the language's grammar for the targeted construct (every import/export/registration production, one fixture each) instead of fixture-driven evasion rounds; (v) classifier failure fails the gate closed — no text fallback. Pin accepted residuals (dead branch inside a running callback, statically empty parameterized cases) as explicit boundary self-tests. **Reviewer action**: for every new test/gate in the diff, ask "what mutation makes this red?" and confirm it exists; for every authored check script, grep the CI/pre-push config for an invocation. Companion to RT4 (race vacuous-pass) and RT6 (new exports untested) — RT7 generalizes "prove the check can fail" to non-race, non-additive, and gate-wiring cases. **Mechanical detection (partial)**: `bash ~/.claude/hooks/check-orphaned-checks.sh [base-ref]` flags shape (b) — authored check scripts with no invocation in CI/pre-push config. Shapes (a) and (c) are human-review. | Major (Critical when the blind gate guards a security invariant) |
