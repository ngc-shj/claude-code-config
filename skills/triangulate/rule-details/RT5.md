# RT5 — Test call-path must include the production primitive

This file contains the full normative procedure for `RT5`. The compact row in `../common-rules.md` is only its routing summary.

## Full rule

| ID | Pattern | Procedure | Severity if missed |
|---|---|---|---|
| RT5 | Test call-path must include the production primitive | When a test claims to verify a production primitive (advisory lock, distributed lock, idempotency key, retry budget, rate limiter, token revocation, audit emit, cache invalidation, signature verification, RLS predicate, transaction-isolation guard, etc.), the test's call path MUST include the call site where the primitive is actually acquired/applied/enforced in production code. A test that calls a helper directly while the primitive is acquired one layer up (e.g., test calls `processMutation()` but the lock is `pg_advisory_xact_lock` taken in the route handler `POST /resource`) does NOT verify the primitive — it verifies a side-effect that may coincidentally hold without the primitive working. **Procedure**: for each test whose name or assertion implies a production-primitive guarantee, trace (a) the primitive's call site in production code, then (b) the test's call path (helper / route / fetch / message handler / SDK call). If (b) does not include (a), the test is mis-named at minimum and silently broken at worst — the assertion may pass under conditions unrelated to the primitive. Materially **worse than RT4** (vacuous-pass on cardinality) because the test name LIES about what it covers: a reviewer reading "advisory-lock prevents double-success" sees a green test and trusts the lock works, when the test never exercised the lock at all. **Severity escalation**: Critical when the production primitive is security-relevant (auth, authz, rate-limit, signature verification, RLS, idempotency on a security-state mutation); Major otherwise. **Detection**: this is a human-review check — grep heuristics produce too many false positives. The reviewer reads the test's `import` / first lines and compares against the production primitive's location; mismatches are findings. **Module-global identity-guard sub-clause**: when a test patches a module-level singleton (a cache, registry, store, or client instantiated once at module scope and referenced by production code), the test MUST assert the patched object **is** the same instance the production code references (`code.singleton is mock` / `expect(prod.store).toBe(mock)`), not merely that it is type-compatible. A type-compatible patch that is not the same object lets production keep using the original default instance while the test exercises a disconnected mock — a silent-pass exactly like calling the wrong call path. | Critical (security-relevant primitive) / Major (otherwise) |
