# RT4 — Race-test vacuous-pass guard

This file contains the full normative procedure for `RT4`. The compact row in `../common-rules.md` is only its routing summary.

## Full rule

| ID | Pattern | Procedure | Severity if missed |
|---|---|---|---|
| RT4 | Race-test vacuous-pass guard | Race-style tests with cardinality assertions (e.g., "no double-success", "exactly K winners", "at most N successes") MUST additionally assert that **both branches of the contested outcome occurred at least once** during the test run. A test asserting `bothSucceeded === 0` without asserting `successes > 0` is vacuously satisfiable: if a setup misconfiguration causes every iteration to short-circuit (e.g., RLS denies all rows, lock is never contended, the contested resource never exists), the cardinality assertion still passes and the race-condition bug ships to production. **Required pattern** (illustrative — adapt assertion library to project): `expect(successes + failures).toBe(N)` (cardinality, existing) PLUS `expect(successes).toBeGreaterThan(0)` (race window opened) PLUS `expect(failures).toBeGreaterThan(0)` (contention occurred). The "guard" assertions name the precondition the cardinality assertion silently depends on; without them, a green test cannot distinguish "race correctly serialized" from "race never happened". Apply to: optimistic-locking tests, idempotency-key tests, distributed-lock tests, queue dedup tests, exclusive-write tests, leader-election tests, any test whose intent is "exactly one of N concurrent attempts wins". **Mechanical detection**: `bash ~/.claude/hooks/check-race-vacuous-guard.sh [base-ref]` walks Jest / Vitest `it(...)` / `test(...)` blocks per changed test file (paren-balance tracking), flags Critical when a block contains a concurrency primitive (`Promise.all` / `allSettled` / `race` / `any`) AND a zero-cardinality assertion (`.toBe(0)` / `.toEqual(0)` / `.toStrictEqual(0)`) AND no lower-bound guard (`.toBeGreaterThan(N)` / `.toBeGreaterThanOrEqual(N)` / `.not.toBe(0)` / `.toBeTruthy()`). The cardinality line must be in the diff `+` set to flag (existing well-guarded tests are not re-flagged on adjacent edits). v1 covers `it`/`test` blocks in TS / JS / TSX / JSX test files; pytest / `t.Run` / RSpec are out of scope. Cardinality values other than 0 (e.g. `.toBe(1)` for "exactly one winner") are precision-trade-offs and not detected in v1; opt in via `EXTRA_CARDINALITY_RE`. Project-specific concurrency helpers and custom matchers extend via `EXTRA_CONCURRENCY_RE` / `EXTRA_GUARD_RE`. | Critical |
